# 端到端（E2E）测试场景设计

## 1. 文档控制

| 项目 | 内容 |
|---|---|
| 文档编号 | TMS-E2E-001 |
| 编制日期 | 2026-06-15 |
| 状态 | 场景设计完成，待真实浏览器执行 |

## 2. 测试目标

在真实浏览器环境中验证三角色（顾客→商家→骑手）完整业务流程，覆盖页面交互、路由跳转、WebSocket 实时通知和状态同步。

## 3. 工具与框架

| 工具 | 用途 |
|---|---|
| Playwright | 浏览器自动化，支持 Chromium/Firefox/WebKit |
| Python / Node.js | 测试脚本语言，推荐 Python + `pytest-playwright` |

## 4. 测试场景

### E2E-01：全流程闭环（下单→接单→抢单→送达）

| 步骤 | 角色 | 操作 | 预期 |
|---|---|---|---|
| 1 | 商家 | 登录 → 添加菜品"红烧肉"¥28 | 菜品列表显示"红烧肉" |
| 2 | 顾客 | 登录 → 浏览菜品 → 看到"红烧肉" | 菜品卡片显示 ¥28 |
| 3 | 顾客 | 添加地址"软件园1号楼 138xxxx" | 地址列表显示 |
| 4 | 顾客 | 下单 → 选"红烧肉" x2 | 提示"下单成功" |
| 5 | 商家 | WebSocket 收到"新订单"通知 | 弹窗提醒 |
| 6 | 商家 | 订单列表 → 接单 | 状态 0→1 |
| 7 | 骑手 | 登录 → 可抢订单列表 | 看到该订单 |
| 8 | 骑手 | 点击"抢单" | 抢单成功，状态 1→2 |
| 9 | 骑手 | 我的订单 → 确认送达 | 状态 2→3，骑手恢复空闲 |
| 10 | 顾客 | 历史订单 → 查看状态 | 状态显示"已完成" |

### E2E-02：商家拒单流程

| 步骤 | 角色 | 操作 | 预期 |
|---|---|---|---|
| 1 | 顾客 | 下单 | 成功 |
| 2 | 商家 | 订单列表 → 拒单 | 状态 0→5 |
| 3 | 顾客 | 历史订单 | 状态显示"商家已拒单" |
| 4 | 骑手 | 可抢订单列表 | 不显示该订单 |

### E2E-03：顾客取消订单

| 步骤 | 角色 | 操作 | 预期 |
|---|---|---|---|
| 1 | 顾客 | 下单 | 成功（状态 0） |
| 2 | 顾客 | 历史订单 → 取消 | 状态 0→4 |
| 3 | 商家 | 订单列表 | 状态"客户已取消"，不可操作 |

### E2E-04：越权访问拦截

| 步骤 | 角色 | 操作 | 预期 |
|---|---|---|---|
| 1 | 顾客 | 登录后直接访问 `/merchant` | 路由守卫重定向到登录页 |
| 2 | 未登录 | 直接访问 `/customer` | 重定向到登录页 |
| 3 | 商家 | 尝试获取顾客地址 API | 403 Forbidden |

### E2E-05：并发抢单 UI 验证

| 步骤 | 操作 | 预期 |
|---|---|---|
| 1 | 两个骑手窗口同时打开可抢订单 | 两个窗口都显示同一订单 |
| 2 | 骑手 A 点击抢单 | 成功，状态更新 |
| 3 | 骑手 B 刷新或等待实时更新 | 订单从列表中消失 |

## 5. 执行说明

本 E2E 方案作为交付物提供，实际浏览器执行待后续排期。当前 API 层自动化测试（34 用例）已覆盖全部后端业务逻辑，E2E 主要用于验证前端 UI 交互和端到端数据一致性。
