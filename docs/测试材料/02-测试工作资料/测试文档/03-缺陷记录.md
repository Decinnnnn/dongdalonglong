# 缺陷记录

## DEF-001 并发抢单缺少原子条件更新

| 字段 | 内容 |
|---|---|
| 严重级别 | P1 / 高 |
| 状态 | **已关闭** ✅ |
| 关联需求 | R-DATA-02、R-COU-01 |
| 位置 | `backend/app/routers/courier.py` 的 `pickup_order` |
| 检出方式 | 自动化并发测试 |
| 修复日期 | 2026-06-15 |
| 修复人 | 开发团队 |
| 验证人 | 李艺涵（成员 C） |

**问题描述：** 抢单流程先读取订单状态和骑手状态，再修改并提交；查询没有行锁，也没有使用 `UPDATE ... WHERE status=1 AND courier_id IS NULL` 的原子条件更新。两个并发事务可能都通过校验并返回成功，最终后提交者覆盖 `courier_id`，同时两个骑手都变为忙碌。

**复现：**

1. 创建一个状态为 1、无骑手的订单。
2. 两个空闲骑手并发调用 `PUT /api/courier/order/{id}/pickup`。
3. 观察两个响应和最终骑手状态。

**期望：** 仅一个请求成功，另一个返回 400/409；只有获胜骑手变为忙碌。

**修复方案：** 将 `pickup_order` 中的先读后写改为带状态条件的原子 `UPDATE`：
```python
# 原子条件更新：仅当订单 status=1 且无骑手时才更新
result = await db.execute(
    update(Orders)
    .where(
        Orders.id == order_id,
        Orders.status == 1,
        Orders.courier_id.is_(None),
    )
    .values(status=2, courier_id=courier.id, order_time=datetime.now())
)
if result.rowcount == 0:
    # 已被抢走或状态已变更
    raise HTTPException(...)
```
- 检查 `result.rowcount`：0 表示已被抢走或状态已变更
- 骑手状态更新与订单更新位于同一 `commit`

**回归结果：** `test_only_one_courier_can_win_concurrent_pickup` 由 `xfail` 改为 `passed`，10 次并发重复执行均仅一人成功（返回 200 vs 400），获胜骑手状态正确变为忙碌，无数据残留。

## DEF-002 bcrypt 依赖未锁定导致注册和登录不可用

| 字段 | 内容 |
|---|---|
| 严重级别 | P0 / 阻断 |
| 状态 | **已关闭** ✅ |
| 关联需求 | R-AUTH-01 |
| 位置 | `backend/requirements.txt`、`backend/app/utils.py` |
| 检出方式 | 首轮自动化执行 |
| 修复日期 | 2026-06-15 |
| 修复人 | 开发团队 |
| 验证人 | 李艺涵（成员 C） |

**问题描述：** `passlib[bcrypt]==1.7.4` 会安装当前最新 `bcrypt`。在 `bcrypt 5.0.0` 环境中，Passlib 初始化后端时触发 `ValueError: password cannot be longer than 72 bytes`，导致所有注册请求异常退出，登录也不可用。

**修复方案：** 在 `requirements.txt` 中新增 `bcrypt==4.0.1`，锁定兼容版本。`passlib[bcrypt]` 检测到已有 `bcrypt==4.0.1` 将跳过安装更高版本。

**回归结果：** 三端注册/登录全部 6 个测试用例通过。从干净环境 `pip install -r requirements.txt` 后注册和登录均正常。
